# Design system portfolio checklist

Use before publishing or interviewing.

## System depth

- [ ] Tokens: taxonomy diagram or table (primitive → semantic → component)
- [ ] At least one component deep-dive (states, a11y, API)
- [ ] At least one pattern (when / when not, recipe)

## Operations

- [ ] Contribution workflow (diagram or steps)
- [ ] Governance: who decides, review gates, deprecation policy
- [ ] Release / changelog story
- [ ] Doc template described (Design vs Dev or usage/style/code/a11y)

## Enablement

- [ ] Onboarding or starter path
- [ ] Office hours / workshops / Slack support (if applicable)
- [ ] Migration or rollout plan referenced

## Evidence

- [ ] Every strong claim has artifact: screenshot, snippet, metric, or link
- [ ] Numbers labeled: measured / estimate / directional / proxy
- [ ] Private work marked **Private — available on request** + sanitized visuals if possible

## Accessibility

- [ ] Heading order logical on each page
- [ ] Images have alt text (describe what the screenshot proves)
- [ ] Link text meaningful (“Read Aurora case study” not “click here”)

## Integration

- [ ] Product usage screenshot or placeholder with caption
- [ ] Figma / Storybook / repo links or placeholders

## Story quality

- [ ] No jargon without one-line definition
- [ ] Outcomes specific (“34 components documented” vs “better consistency”)
- [ ] Reflections: tradeoffs + failure + redo

## Site hygiene

- [ ] `/design-systems` summarizes scope and links 2–4 case studies
- [ ] Resume page has 5–8 highlight bullets (artifact + scope + outcome)
- [ ] PDF resume path verified if linked
